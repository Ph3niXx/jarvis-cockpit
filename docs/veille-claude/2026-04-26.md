## jarvis_applicable

### [HIGH · M] Memory pour Claude Managed Agents : public beta, filesystem-mounted, audit log

**Source** — [Anthropic blog](https://claude.com/blog/claude-managed-agents-memory) · [docs](https://platform.claude.com/docs/en/managed-agents/memory)

Anthropic a sorti le 23 avril Memory pour Managed Agents en public beta (header `managed-agents-2026-04-01`). Architecture : memory store = collection workspace-scoped de fichiers texte, montée directement sur le filesystem du sandbox de l'agent. Plusieurs agents peuvent lire/écrire en concurrent, chaque mutation crée une version immuable, redact possible pour scrubber des PII a posteriori. Scopes : org-wide read-only, per-user read-write, partage cross-agent. Adoptants annoncés : Netflix, Rakuten, Wisedocs, Ando.

> **How to apply** : architecture concurrente directe de la mémoire structurée Jarvis (tables `profile_facts` + `entities` + `jarvis_conversations` peuplées par `nightly_learner.py`). Avantages d'un switch : audit log natif, versioning, redact, pas de pgvector à maintenir, scalable. Désavantages : verrouillage Anthropic, runtime billing 0,08$/h, pas de RAG sémantique côté serveur (memory = fichiers texte simples). Test pilote possible : créer un memory store "user_profile" en parallèle de la table Supabase pendant 2 semaines, comparer la pertinence d'extraction de faits de Claude Haiku via memory vs. via system prompt injecté.

> **Trend** : 4ᵉ produit Anthropic en 6 mois qui internalise une couche d'infra agent qu'on faisait en glue maison (Routines, Scheduled Tasks, Managed Agents, désormais Memory).

### [HIGH · M] Claude Design : prompt → prototype design avec handoff Claude Code

**Source** — [Anthropic Labs](https://www.anthropic.com/news/claude-design-anthropic-labs) · [TechCrunch](https://techcrunch.com/2026/04/17/anthropic-launches-claude-design-a-new-product-for-creating-quick-visuals/)

Lancé le 17 avril, research preview pour Pro/Max/Team/Enterprise. Powered by Opus 4.7 (vision améliorée 3x, jusqu'à 2576px sur le côté long). Inputs acceptés : prompts texte, codebases existantes, exports Figma, PDFs, screenshots, briefs voice, transcripts de meetings. Outputs : design systems, prototypes interactifs, slides, one-pagers, websites. Différenciateur : un handoff bundle exportable d'un clic vers Claude Code, qui fait la boucle exploration → prototype → prod sans changer d'écosystème. Stock Figma a perdu 5-7% le jour du lancement.

> **How to apply** : pour le cockpit, la stack actuelle est React 18 + babel standalone + CSS variables par thème (Dawn/Obsidian/Atlas). Test pilote : prendre un panel récent qui a souffert (ex : panel Forme avec ses 4 sous-vues mergées) et demander à Claude Design un re-mockup à partir du screenshot actuel + un brief produit. Comparer le bundle handoff au panel existant et estimer combien d'allers-retours on aurait gagnés. À calibrer ensuite sur le prochain nouveau panel (ex : si on ajoute "Cockpit Finance" plus tard).

> **Trend** : Anthropic absorbe la couche design upstream du code — direct concurrent Figma sur les briefs simples.

### [MEDIUM · S] Claude Code Ultraplan : draft de plan dans le cloud, exécution locale ou cloud

**Source** — [Claude Code Docs](https://code.claude.com/docs/en/ultraplan) · [Better Stack](https://betterstack.com/community/guides/ai/claude-code-ultraplan/)

Ultraplan = plan mode déporté en cloud session (CCR Anthropic, contexte 1M tokens via Sonnet 4.6). Tu lances depuis ton CLI local, Claude rédige le plan dans une UI web pendant que ton terminal reste libre. UI : commentaires inline, réactions emoji approve/revise, sidebar outline structurée. Au bout : approve → Claude implémente directement dans le cloud (PR auto-ouverte) OU "teleport back" du plan vers le terminal local pour exécution avec accès complet à l'environnement. Requiert Claude Code v2.1.101+, repo GitHub, plan Pro ou Max. Research preview, lancé en semaine 15 (6-10 avril).

> **How to apply** : à tester sur la prochaine grosse refacto du cockpit qui mérite vraiment un plan revu (ex : consolidation des 5 panels Veille en un sous-routeur unique, ou refacto du data layer Tier 1/Tier 2). Workflow : `/ultraplan refacto data-loader.js Tier 1/Tier 2` → review web → teleport back → exécution locale avec service worker offline. Avantage clé sur le workflow actuel : plan asynchrone (je ferme l'onglet pendant qu'il pense), commentaires inline (vs. tout réécrire le prompt à chaque tour), 1M tokens (peut digérer toute l'arbo `cockpit/`).

> **Trend** : 2ᵉ produit cloud-first de Claude Code en 30 jours après Routines — l'inflexion "thin terminal client + heavy cloud session" se confirme.

## complementary_tools

### [MEDIUM · S] ant CLI : client Claude API en ligne de commande, output formatable

**Source** — [Claude API Docs](https://platform.claude.com/docs/en/api/sdks/cli) · [GitHub](https://github.com/anthropics/anthropic-cli)

Anthropic a publié `ant`, CLI Go qui expose chaque ressource de l'API Claude comme sous-commande, avec inputs YAML/JSON, file inlining via `@path`, et extraction de réponse via `--transform`. Comparé à un curl + jq + bash, l'auteur prétend "plus pratique pour exploration interactive et automation". Requiert Go 1.22+. Install : `go install github.com/anthropics/anthropic-cli/cmd/ant@latest`.

> **How to install / apply** : pour MOI, cas d'usage spécifique = remplacer les blocs Python dans `weekly_analysis.py` qui font des `requests.post` manuels vers `messages.anthropic.com` quand un script jetable suffit. Exemple : un quick-eval "donne-moi une note de 1-10 sur ce brief pour le panel X" depuis un terminal, sans booter un venv. Pas un remplaçant de `weekly_analysis.py` (logique métier + CostTracker), mais un outil de scripting léger pour tester un prompt Claude rapidement.

> **Trend** : Anthropic standardise sa surface de scripting au-delà du SDK Python/TS. Si `ant` devient mainstream, c'est un meilleur premier pas que SDK pour les power-users.

### [MEDIUM · XS] Token counter Simon Willison : Opus 4.7 pèse 1.46x les tokens d'Opus 4.6 sur du texte

**Source** — [Simon Willison](https://simonwillison.net/2026/apr/20/claude-token-counts/) · [Token counter tool](https://tools.simonwillison.net/claude-token-counter)

Simon Willison a outillé son token counter pour comparer plusieurs modèles Claude. Sur le system prompt d'Opus 4.7 lui-même, le tokenizer 4.7 produit 7335 tokens contre 5039 pour 4.6, soit 1.46x. Sur du PDF text-heavy : 1.08x seulement. Sur des images, hausse essentiellement liée à la nouvelle résolution max 2576px (et non au tokenizer). Conclusion utile : la hausse de coût réelle dépend du type de contenu, pas un blanket +35%.

> **How to apply** : confirme la mise en garde de la semaine dernière sur le CostTracker du pipeline `weekly_analysis`. Action concrète : avant de basculer un seul prompt Haiku → Opus 4.7, prendre un sample de 10 prompts récents du pipeline et les passer dans le token counter pour mesurer le facteur empirique sur MON contenu (briefs articles français + extraits user_profile). Si ratio > 1.3x → garder Haiku pour le volume, n'envoyer qu'un meta-prompt synthèse à Opus 4.7.

> **Trend** : Simon Willison sort un outil utile par incident — pattern qu'on peut réutiliser pour le cockpit (mini-tools React isolés branchés à l'API).

### [MEDIUM · S] Claude Code Week 15 : /team-onboarding + /autofix-pr + Monitor tool

**Source** — [Claude Code Docs Week 15](https://code.claude.com/docs/en/whats-new/2026-w15)

> **How to apply** : pour le cockpit, `/autofix-pr` est tentant pour les workflows CI bloquants (validate-spec, lint-specs, validate-arch). Si oui → gain énorme sur les PRs futures qui touchent un panel ET sa spec. Pour `/team-onboarding`, peu pertinent en solo, mais pourrait servir de "self-onboarding" après une pause longue (ré-apprendre mes propres conventions).

> **Trend** : Claude Code converge vers un workflow "fire-and-forget" où on lance le travail et Claude tient le CI vert tout seul.

### [MEDIUM · XS] Snyk ToxicSkills v2 : 36% des skills publics ont une faille, 71 malveillants

**Source** — [Snyk research](https://snyk.io/blog/toxicskills-malicious-ai-agent-skills-clawhub/) · [Cybersecuritywaala](https://cybersecuritywaala.com/blog/71-malicious-claude-skills-found/)

Snyk a scanné 3984 skills depuis ClawHub et skills.sh : 36.82% ont au moins une faille de sécurité, 13.4% ont des issues critiques, 76 payloads malicieux confirmés (credential theft, backdoor, data exfiltration), 71 skills malveillants distincts identifiés. Cause structurelle : skills exécutés avec les pleines permissions de l'agent qui les charge ; barrière de publication = un fichier `SKILL.md` + un compte GitHub d'1 semaine, sans signing ni review. Recommandation : rotation credentials si tu as installé un skill qui touche API keys / cloud creds, audit `MEMORY.md` / `SOUL.md` du projet.

> **How to apply** : aujourd'hui, le cockpit n'installe AUCUN skill tiers (skills officiels Anthropic + skills user maison uniquement). Action préventive : si jamais on installe `evals-skills` de Hamel (item de la semaine dernière) ou un autre, le faire en sandbox isolé d'abord (un repo de test), pas direct dans `jarvis-cockpit`. Et garder `ANTHROPIC_API_KEY` jamais dans un skill mais en env var.

> **Trend** : marketplace Skills suit la trajectoire npm/PyPI — distribution easy, sécurité à la traîne. À surveiller : Anthropic va probablement finir par signer les skills officiels.
