### S1 — Réparer `get_chunks_count()` dans `status_generator.py`

**Pourquoi maintenant** : sert directement la North Star. P34 a été parké le 26/04 avec condition de réveil "Snapshot Jarvis Lab consulté ≥ 3x/sem OU feature dépend de cette métrique" — **`jarvis-lab` est le 4ème panel le plus consulté cette semaine (27 ouvertures)**. Condition largement remplie. Coût marginal minuscule (~30 min).
**Diagnostic** : `jarvis/status_generator.py:107-132` (`get_chunks_count()`). La fonction fait `GET /rest/v1/memories_vectors?select=id&limit=0` avec `Prefer: count=exact` puis lit `Content-Range` pour extraire le total. PostgREST renvoie `Content-Range: */N` ou `0-N-1/N` selon la version, et avec `limit=0` certaines versions renvoient `*/*` (count non résolu) — ce qui explique le `0` reporté en log alors que la table contient 1000+ vecteurs. Le pattern canonique PostgREST est `Range: 0-0` HEADER + `Prefer: count=exact`, pas `?limit=0` query param.
**Définition de "fait"** : après modification, `python jarvis/status_generator.py` log `Chunks: N` avec N ≥ 1000 (vrai compte). Le panel Jarvis Lab affiche le bon chiffre à la prochaine ouverture.
